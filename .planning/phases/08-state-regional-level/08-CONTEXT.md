<decisions>
## Implementation Decisions

### Voice & Authority
- Claude's discretion on tone (professional peer vs technical expert) based on specific pathway
- Reference regulatory frameworks where relevant to the role, not uniformly
- Claude judges terminology definitions by pathway - technical paths assume more baseline knowledge
- Claude adjusts timeline estimates based on project type (regulatory work may span months)
- Keep prompts qualitatively focused - let project ideas introduce quantitative work
- Claude decides when to name specific stakeholders (utility commissioners, PUC staff)
- Claude calibrates follow-up prompt sophistication to pathway entry requirements
- Claude references utility types (IOU, municipal, co-op) when distinction matters
- Keep prompts state-agnostic - focus on transferable concepts across states

### Certification Focus
- Claude integrates certifications naturally where they add value (prompt context and/or project ideas)
- Claude calibrates whether certifications are presented as required vs optional accelerators by role
- Claude determines certification relevance pathway by pathway
- Claude includes cost context when it's a significant barrier
- **Key addition:** When high-cost items come up, include information about grants, assistance programs, and how AI can help navigate those

### Regulatory Framing
- Balance utility regulation AND building code content equally across the 25 pathways
- Claude decides when to name specific programs (RPS, EERS, BPS) vs keep general
- Use real regulatory examples rather than hypothetical scenarios
- Keep employer references implicit - let work context imply the career landscape

### Prompt Patterns
- Create ALL NEW patterns for this level (not reusing Phase 6-7 patterns)
- Claude develops patterns that fit specific pathways - could include regulatory document patterns, career milestone patterns, or technical skill patterns
- 5 distinct patterns per card, no adjacent duplicates within card (same structure as Phase 7)
- Claude decides whether to include GSD workflow reference (if it fits naturally)

</decisions>
